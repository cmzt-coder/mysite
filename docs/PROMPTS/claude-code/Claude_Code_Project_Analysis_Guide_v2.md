# 使用 Claude Code 快速拆解任意开源项目架构与二次开发通用指南

本指南旨在帮助你利用 Claude Code 强大的本地代码感知与 Agent 能力，通过一套**通用的渐进式交互框架（Iterative Prompting Framework）**，在极短时间内快速熟悉任何一门陌生语言、任意业务领域的开源项目，厘清核心功能逻辑，并为后续的二次开发、重构或功能迁移打下坚实基础。

---

## 📌 核心思想：渐进式交互四步法

面对动辄成百上千个文件的未知代码库，切忌一开始就让 AI “通读并解释所有代码”。最有效率的做法是由表及里、由面到点地进行提问：

```
[ 阶段一：宏观拓扑 ] ──> [ 阶段二：垂直链路 ] ──> [ 阶段三：水平扩展 ] ──> [ 阶段四：资产固化 ]
(看清地图与技术栈)       (追踪核心业务数据流)       (寻找二开与插桩位)       (生成本地 AI 导航大脑)
```

---

## 🛠️ 通用实战命令模板

请在终端进入你的目标项目根目录，输入 `claude` 启动交互界面后，依次发送以下指令：

### 阶段一：宏观认知（技术栈与目录拓扑）
**目的：** 构建项目的全局地图，了解项目的骨架、核心技术栈、打包构建机制以及模块划分。

👉 **发送给 Claude 的命令：**
```text
请帮我整体扫描当前项目的目录结构与核心配置文件（如 package.json / go.mod / pom.xml / pyproject.toml / Cargo.toml 等）。
请分析并告诉我：
1. 本项目采用的核心技术栈、运行时环境及关键的第三方依赖。
2. 给出高层级的模块/文件夹划分概述，用清晰的层级结构（如 Markdown 列表或表格）说明每个核心文件夹在架构中所扮演的角色。
3. 该项目的入口文件（Entry Point）或主启动类在哪里？
```

---

### 阶段二：垂直链路追踪（核心业务数据流）
**目的：** 不去通读所有边缘代码，而是像“切蛋糕”一样，切出一条最核心的业务主线，让 Claude 带你从头到尾走一遍。

> 💡 **使用提示：** 请根据你所爬取/下载的项目类型，选择或修改下方对应的【行业通用模版】发送给 Claude：

*   **模版 A：Web / 后端 API 项目（如电商、内容管理系统、微服务等）**
    ```text
    我想深入理解这个项目的核心业务流程。请帮我追踪一个最典型的数据请求流向（例如：【用户提交订单 / 用户注册登录】）。
    请从最外层的路由、API 接口/控制器（Controller/Handler）开始，向内追踪它调用了哪些中间件（Middleware）、服务层（Service/UseCase）、领域模型（Domain），以及最终如何通过 ORM 或数据访问层（DAO/Repository）写入数据库。请列出涉及的关键文件路径、类名与核心函数。
    ```

*   **模版 B：前端 / 移动端应用（如 React/Vue 单页应用、Flutter、iOS/Android）**
    ```text
    我想理解这个前端项目的状态管理和页面渲染逻辑。请帮我追踪一个典型用户交互（例如：【用户点击加入购物车并引发顶栏数量更新】）的完整生命周期。
    请说明它如何从 UI 组件触发事件，如何通过状态管理库（如 Redux, Pinia, Vuex 或 Context）进行状态派发，又是如何调用封装好的 API 客户端进行后端通信，并最终完成界面重绘的。请指出关键的组件文件、状态定义及 Action 函数。
```

*   **模版 C：CLI 工具 / 系统级软件 / 数据管道（如自动化脚本、ETL 工具、编译器）**
    ```text
    我想理清这个工具的执行核心。请帮我追踪一次完整运行（例如：【解析一行命令行输入并执行核心任务 / 处理一个输入文件】）的内部调用链路。
    请从命令行参数解析（CLI Argument Parsing）开始，展示它如何将输入转化为内部配置对象，调用了哪些核心执行器（Executor/Runner），如何进行错误处理与日志记录，并最终输出结果。请列出涉及的核心函数和模块路径。
    ```

---

### 阶段三：水平扩展评估（二次开发扩展点）
**目的：** 明确代码的解耦程度，为你的定制化开发、功能插桩或魔改寻找最佳落脚点。

👉 **发送给 Claude 的命令：**
```text
我的最终目的是基于当前项目进行二次开发与定制扩展。请帮我评估当前代码的解耦程度和扩展性：
1. 如果我想要【添加一个新的定制功能模块 / 扩展一种新的插件或驱动支持 / 修改现有的某个核心行为】，我应该在哪些目录下添加新代码？
2. 当前项目是否采用了某些设计模式（如策略模式、工厂模式、观察者模式）或定义了特定的接口（Interface）/ 基类（Base Class）供开发者继承扩展？
3. 在进行二次开发时，有哪些全局的公共工具类（Utils）、上下文（Context）或配置管理类（Config）是我可以直接复用的？
```

---

### 阶段四：资产固化（生成项目专属的 AI 导航大脑）
**目的：** 将刚才所有的问答与推导结果，固化为项目根目录下的持久化文件。这不仅方便你人类阅读，更为后续让 Claude Code 自动帮你写代码打下伏笔。

👉 **发送给 Claude 的命令：**
```text
请结合刚才我们所有的分析与你的推导结果，利用你的文件写入工具，在项目根目录下为我生成和完善两个关键的 Markdown 文档：

1. 创建 `ARCHITECTURE.md`：详细记录该项目的整体架构图景、模块职责划分、核心业务的调用栈/数据流向，以及二次开发的切入点说明。
2. 创建或完善 `CLAUDE.md`：严格根据 Claude Code 的官方最佳实践，该文件必须包含：
   - 项目简介与核心技术栈概述；
   - 核心模块与文件拓扑；
   - 本地常用的开发、构建、测试、格式化与运行命令（请确保命令真实可用）；
   - 项目的代码风格约束（如命名规范、错误处理倾向、注释规范）与二次开发扩展设计原则。
```

## 🚀 进阶高级技巧：如何把 Claude Code 当作资深架构师？

### 1. 差异与变更分析（Git 集成感知）
Claude Code 能够读取本地 Git 状态。如果你想通过查看前人的 Commit 记录或不同分支来学习某个特性的实现，可以直接对它说：
> *“请对比当前分支与 main 分支的差异，告诉我作者为了实现 `XXX` 特性，一共修改了哪几个文件？请为我剖析他的修改思路。”*

### 2. 局部破冰：利用 `grep` 挖掘隐蔽逻辑
当遇到某些高度抽象（如依赖注入、事件驱动、异步消息队列）的项目时，静态阅读可能难辨调用关系。你可以直接调动 Claude Code 的 Agent 能力在本地进行深度搜索：
> *“这个项目里似乎用了异步事件通知。请帮我用 grep 搜索所有的事件发布（Publish/Emit）和事件监听（Subscribe/Listen/On）位置，梳理出一张事件流转图，告诉我 `X事件` 触发后，到底有哪些地方在响应。”*

### 3. 测试驱动理解法（Test-Driven Understanding）
了解一个函数或类怎么用，最好的方式是看它的单元测试。如果代码本身缺乏注释，你可以命令 Claude：
> *“请带我阅读 `tests/` 或 `__tests__/` 目录下针对核心模块 `XXX` 的测试用例。结合测试用例中的 Assert（断言）和 Mock 数据，反向为我解释这个核心模块的边界条件和业务规则是什么。”*

### 4. 终极联动：让 `CLAUDE.md` 自动化你的下一次 Copilot 体验
步骤四中生成的 `CLAUDE.md` 具有神奇的魔力。它是 Claude Code 官方支持的系统级上下文锚定文件。
当你完成了架构梳理，几天后想要真正开始写二次开发代码时，你只需再次进入目录并输入：
`claude "请帮我实现一个满足 XXX 业务的自定义扩展插件"`
Claude Code 在行动前会**自动、隐式地读取**你之前让它生成的 `CLAUDE.md`。它会瞬间记起这个项目的全部命令、目录规范和代码风格，从而生成**完全符合该开源项目原生脉络、零违和感**的高质量代码。
